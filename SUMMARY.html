THE FLAKY NETWORK

architecture summary — forward-only XOR FEC sender/receiver in C

A live-audio stream — one 160-byte frame every 20 ms — has to cross a UDP path that drops, delays, reorders, and duplicates packets, and still get played out on a steady clock at the far end. This submission's answer is: never wait for a second round trip. Send a little extra data ahead of time so the receiver can reconstruct a lost frame the moment it notices one is missing, instead of asking the sender to resend it.

3 · The mechanism: forward-only XOR FEC

Frames are grouped contiguously into blocks of G (default 3, overridable with env FEC_G for experiments). Every frame is forwarded to the relay the instant it arrives from the harness source — no batching, no held-back scheduling. After the sender has sent the G-th frame of a block, it also sends one parity packet: the byte-wise XOR of that block's G payloads.

On the receiver side, every arriving DATA or PARITY packet is stored and then triggers a check: "does this block now have exactly one member missing?" If yes, and the parity has arrived, the missing payload is reconstructed by XOR-ing the parity with the other G−1 present payloads — algebraically exact, since XOR is its own inverse. The recovered frame is forwarded to the harness player immediately, in the same fixed seq(u32be) + 160B payload format it expects on port 47020.

There is no reordering or hold-and-wait buffer on the receiver. The harness player already scores "first correct arrival before its deadline," so the only thing worth doing locally is (a) forward the instant we have a frame, and (b) make "having a frame" possible slightly earlier for frames that were actually lost, via the parity. The only added latency this introduces is intrinsic to the FEC group size itself: a lost frame can't be reconstructed until the block's last member and its parity have both gone out, i.e. up to (G−1)×20ms after the loss — 40 ms for G=3.

4 · Why FEC and not retransmission

"What does a resend cost in time when the request AND the reply each cross the hostile network?"

A NACK-based scheme needs the receiver to notice a gap, ask the sender for it (crossing the relay once), and get the resend back (crossing the relay again). Under profile B's 20–80 ms per-hop jitter, that round trip alone can burn well over 100 ms before the resend even leaves the sender — more delay budget than the whole FEC scheme costs end-to-end, and the resend request or its reply can itself be dropped, cascading into a second wait. Sending a small amount of redundancy ahead of time, unconditionally, sidesteps this entirely: there is no dependency on anything surviving a second network crossing. The feedback ports (47003/47004) are wired up in the harness but our sender/receiver never use them — down_bytes is 0 in every run, which also leaves the full bandwidth budget available for FEC instead of splitting it with a feedback channel.

5 · Bandwidth budget

One parity packet (166B) protects a whole block of G data packets (165B each), so the overhead ratio is (165G + 166) / (160G) — this is why grouping beats naive duplication: duplicating every frame costs a full extra 1× (≈2.0×, the entire budget with no header room left), while a group of G shares one parity packet's cost across G frames.

7 · Known limitation

A contiguous XOR block recovers at most one lost frame per block. Two or more genuine drops landing in the same block are unrecoverable no matter how much delay is budgeted — this is a hard ceiling from the algebra (one parity equation, one unknown), not a tuning problem. In stress testing beyond the given profiles, independent loss much above ~8% or a real burst-loss pattern (several consecutive drops) both push the residual miss rate over the 1% cap, because bursts concentrate multiple losses inside one block. The natural next step, not built here for time, is interleaved FEC — assigning physically-adjacent frames to different blocks so a time-local burst spreads its damage across several blocks (each losing at most one member) instead of concentrating it in one.

8 · Code structure

  • sender.c (~110 lines) — binds 47010, maintains a small G-slot rolling buffer for the block currently being formed, forwards every DATA packet immediately, emits one PARITY packet per completed block. No feedback socket.
  • receiver.c (~185 lines) — binds 47002, sizes its per-sequence bookkeeping arrays from DURATION_S, forwards every DATA packet to 47020 immediately, and on every DATA or PARITY arrival re-checks whether the touched block can now be solved.
  • Byte order: all multi-byte header fields (seq, base_seq) are big-endian on the wire (htonl/ntohl), matching the harness's own framing convention.
  • Buffer ownership: the receiver owns one contiguous calloc'd payload table indexed by seq for the whole run; the sender only ever needs the current block's G slots, since forwarding is immediate and nothing older is ever re-read.
  • Blocking: both processes run a single-threaded blocking recvfrom loop — there is no timer-driven send path (the sender only ever reacts to harness input, never sends spontaneously), so there is no risk of the receive loop blocking at a moment it needed to be doing something else.
